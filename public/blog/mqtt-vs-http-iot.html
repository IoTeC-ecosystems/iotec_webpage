MQTT vs. HTTP en aplicaciones IoT: ¿cuál elegir?

2 Mar 2026 7 min lectura Equipo IoTeC

Dos protocolos, dos filosofías

Cuando se diseña la capa de comunicación de un sistema IoT, la elección del protocolo de transporte tiene implicaciones directas en el consumo de energía, el ancho de banda, la latencia y la complejidad de implementación. Los dos protocolos más frecuentes son MQTT (Message Queuing Telemetry Transport) y HTTP/REST.

La diferencia fundamental es el modelo de comunicación: HTTP sigue un modelo request-response síncrono, mientras que MQTT usa un modelo publish-subscribe asíncrono mediado por un broker.

HTTP/REST en IoT

HTTP es omnipresente: todo desarrollador lo conoce, las bibliotecas están en todos los lenguajes y el ecosistema de herramientas (Postman, curl, etc.) es maduro. Sus ventajas en IoT:

  • Fácil interoperabilidad con backends web y APIs existentes.
  • Soporte nativo de TLS/HTTPS sin configuración adicional.
  • Stateless por diseño, lo que simplifica la escalabilidad de los servidores.

Sus desventajas en entornos IoT de baja potencia o alta frecuencia son significativas:

  • Overhead de encabezados elevado (cientos de bytes por solicitud vs. 2 bytes de cabecera MQTT).
  • Modelo polled: el dispositivo debe preguntar periódicamente si hay comandos nuevos.
  • Conexión TCP por solicitud consume energía en dispositivos con batería.

MQTT y el modelo pub/sub

MQTT fue diseñado en 1999 por IBM para telemetría en oleoductos con conexiones satelitales de baja velocidad. Sus características lo hacen ideal para IoT industrial:

  • Cabecera mínima de 2 bytes para mensajes de control.
  • Tres niveles de QoS (0, 1, 2) que permiten balancear entre velocidad y garantía de entrega según el caso de uso.
  • Will message (LWT): el broker notifica a los suscriptores cuando un cliente se desconecta inesperadamente.
  • Retained messages: un nuevo suscriptor recibe el último valor publicado inmediatamente.
  • Conexión TCP persistente que mantiene sesión y reduce el overhead de handshake.

Comparativa técnica resumida

Característica        MQTT            HTTP/REST
─────────────────────────────────────────────
Modelo                Pub/Sub         Req/Res
Sobrecarga cabecera   2 bytes         ~200-800 bytes
Latencia              Muy baja        Media-alta
Conexión              Persistente     Por solicitud
Soporte bidireccional Nativo          Require polling
Seguridad TLS         Sí (puerto 8883) Sí (HTTPS)
Ideal para            Telemetría      APIs e integrac.

¿Cuándo usar cada uno?

En proyectos IoTeC seguimos esta heurística para elegir el protocolo:

  • Usa MQTT cuando tengas muchos dispositivos enviando datos frecuentemente (más de una lectura por segundo), redes inestables o hardware con restricciones de energía.
  • Usa HTTP/REST para configuración remota de dispositivos, integración con sistemas ERP/MES, o cuando el volumen de mensajes sea bajo y la prioridad sea la simplicidad de integración.
  • Considera combinarlos: MQTT para el flujo de telemetría y HTTP para operaciones de administración y alertas críticas que requieren confirmación.
En nuestra experiencia, migrar de HTTP polling a MQTT en una red de 200 sensores redujo el tráfico de red en un 78 % y la latencia promedio de 800 ms a menos de 50 ms.

Implementación rápida con Python y Paho-MQTT

import paho.mqtt.client as mqtt

BROKER = "broker.iotec.internal"
TOPIC  = "planta/linea-1/temperatura"

def on_message(client, userdata, msg):
    print(f"[{msg.topic}] {msg.payload.decode()}")

client = mqtt.Client()
client.on_message = on_message
client.connect(BROKER, 1883, 60)
client.subscribe(TOPIC, qos=1)
client.loop_forever()

Con menos de 15 líneas de código Python tienes un suscriptor funcional que recibe lecturas de temperatura en tiempo real usando QoS 1 (entrega garantizada al menos una vez).
